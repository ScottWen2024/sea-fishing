# 海钓通 测试实践指南（零基础版）

---

## 一、测试是什么，不是什么

### 测试 = 保险

```
你改了钓鱼指数的公式 → 跑测试 → 自动告诉你"旧逻辑被破坏了"
```

就像开车买保险一样——平时感觉不到，出事了才知道有用。

### 测试 ≠ 保证 Bug 为零

```
测试通过 ≠ 没有 Bug
测试通过 = "你测过的那 22 种情况是对的"
```

没人能测所有情况。测最重要的 20% 场景，覆盖 80% 的潜在问题。

---

## 二、怎么决定测什么（风险优先）

世界上最重要的测试原则：

> **衡量标准不是覆盖率百分比，而是"如果这个错了，损失有多大"。**

应用到海钓通：

| 如果这个错了... | 损失 | 值得测吗 |
|------|:---:|:---:|
| 钓鱼指数算错了 | 钓友相信错误的建议，出海遇险 | ⭐⭐⭐ 必测 |
| GPS 距离算错了 | 推荐了错误的港口 | ⭐⭐⭐ 必测 |
| 和风 API 挂了 | 页面白屏 | ⭐⭐ 要测 |
| 拼船列表顺序不对 | 只是看着不舒服 | ⭐ 可以延后 |
| 按钮颜色偏了 | 不影响功能 | ⭐ 不急 |

## 三、什么是单元测试、集成测试、端到端测试

用一个做饭的比喻：

```
单元测试 = 检查每样食材是不是好的
  "这个鸡蛋新鲜吗？" → 单独检查鸡蛋

集成测试 = 检查几样东西放一起会不会出问题
  "鸡蛋+面粉+油 能做成面糊吗？" → 检查组合

端到端测试 = 试吃整道菜
  "点菜 → 上菜 → 结账 全流程通吗？" → 检查全流程
```

海钓通的对应：

| 类型 | 比喻 | 例子 |
|------|------|------|
| 单元测试 | 检查食材 | `estimateWave(15)` 应该返回 `"0.3-0.6"` |
| 集成测试 | 检查组合 | 和风 API 挂了 → 页面是否降级为模拟数据 |
| E2E | 试吃 | 发起约钓 → 另一个人打开首页 → 能看到 |

## 四、测试写在哪，谁写，谁跑

### 分层负责

| 层级 | 谁写 | 谁跑 | 什么时候 |
|------|:---:|:---:|------|
| 单元测试 | Claude 写 | 你本地 `npm test` | 改代码后 |
| API 测试 | Claude 写 curl 脚本 | 你手动执行 | 发版前 |
| UAT 回归 | checklist.md | 你手动对照 | 发版前 |

### 海钓通的实际分工

```
Claude 负责：
  ✅ 写所有测试代码
  ✅ 设置测试框架
  ✅ 覆盖边界值

你负责：
  ✅ 跑 `npm test`
  ✅ 对照 checklist 手动过一遍
  ✅ 说"这里不太对"
```

## 五、测试用例怎么写

一条测试用例 = 三句话：

```
1. 给我什么输入         →  风速 15km/h
2. 期望得到什么输出     →  浪高 "0.3-0.6"
3. 为什么是这个结果     →  15 在 11-20 区间
```

代码表示：

```javascript
test('风速 15km/h → 0.3-0.6m', () => {
  expect(estimateWave(15)).toBe('0.3-0.6');
});
```

### 测试用例口诀

```
输入 → 输出 → 断言
给什么 → 期望什么 → 检查什么
```

## 六、新人最容易犯的错

### 错误 1：追求 100% 覆盖率

```
❌ "我们必须每条代码都测到"
✅ "测最重要的 7 个函数就够了"
```

### 错误 2：一次想测所有东西

```
❌ "写一个测试覆盖首页所有功能"
✅ "先测 estimateWave，过了再测 getWeatherAdvice"
```

### 错误 3：测试不可靠时还留着

```
❌ "有时候过有时候不过，不管了"
✅ "删掉或者修好，不可靠的测试比没有测试更糟糕"
```

### 错误 4：改完代码不跑测试

```
❌ "我改了一行应该没事"
✅ 每次 git commit 前跑 npm test
```

## 八、日常测试流程（5 分钟版）

```
1. 改完代码
2. cd D:\Fishing project\tests && npm test
3. 看结果：
   - 全绿 ✅ → 提交 git commit
   - 有红的 ❌ → 检查是谁的问题，修好再提交
4. 打开 checklist.md 过一遍今天改的页面
5. 完成
```
